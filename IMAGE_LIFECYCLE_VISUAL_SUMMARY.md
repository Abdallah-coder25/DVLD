## ?? ROOT CAUSE #5: Unsafe File Operation Sequence

```
????????????????????????????????????????????????????????????
? BROKEN: DeleteImage() ? SaveImage()                      ?
????????????????????????????????????????????????????????????
?                                                           ?
?  public static string UpdateImage(old, new)              ?
?  {                                                        ?
?      DeleteImage(old);      ? Deletes here              ?
?      return SaveImage(new); ? Saves here (might fail!)   ?
?  }                                                        ?
?                                                           ?
?  If SaveImage() fails:                                    ?
?    ?? Old: DELETED                                       ?
?    ?? New: FAILED TO COPY                                ?
?    ?? Result: NO IMAGE! Data loss!                       ?
?                                                           ?
????????????????????????????????????????????????????????????

        FIX: SaveImage() FIRST, DeleteImage() SECOND
        
        public static string ReplaceImage(old, new)
        {
            // 1. Try to save new image
            string newPath = SaveImage(new);
            
            // 2. Only if new succeeded, delete old
            if (!string.IsNullOrEmpty(newPath))
            {
                DeleteImage(old);
                return newPath;
            }
            
            // 3. If new failed, keep old
            return old;
        }
```

## ? Key Takeaway

**Original Problem:** Multiple systems (UI, file system, database) were unsynchronized, causing:
- Files locked indefinitely
- Orphaned files left behind
- Data inconsistency
- Duplicate files

**Solution:** Single source of truth with proper sequencing:
1. UI changes (preview without lock)
2. DB changes (only after success)
3. File changes (only after DB confirms)
4. State cleanup (only after all succeed)

**Result:** Clean, consistent, safe image management
